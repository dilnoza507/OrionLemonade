# Архитектурные мысли и варианты реализации

> Заметки по реализации системы «Лимонадный цех»

---

## Стек технологий — варианты

### Вариант A: Классический веб (рекомендуется)
- **Frontend:** React / Vue.js + TypeScript
- **Backend:** Node.js (Express/Fastify) или C# (.NET)
- **БД:** PostgreSQL
- **Отчёты:** библиотека для генерации PDF/Excel

**Плюсы:** доступ с любого устройства, легко обновлять, один сервер
**Минусы:** нужен хостинг, интернет

### Вариант B: Desktop-приложение
- **Стек:** Electron + React или WPF (.NET)
- **БД:** SQLite (локальная) или PostgreSQL (сервер)

**Плюсы:** работает offline, быстрый отклик
**Минусы:** нужно ставить на каждый ПК, обновления сложнее

### Вариант C: Гибрид
- **Backend:** веб-сервер в локальной сети цеха
- **Frontend:** браузерный доступ по IP в локалке
- **БД:** PostgreSQL на том же сервере

**Плюсы:** не зависит от интернета, доступ с любого устройства в сети
**Минусы:** нужна настройка локального сервера

## Приоритеты модулей (рекомендуемый порядок разработки)

1. **Авторизация и роли** — без этого нельзя начать
2. **Склад сырья** — база для всего производства
3. **Рецептуры** — зависит от склада
4. **Производство** — зависит от рецептур и склада
5. **Склад готовой продукции** — результат производства
6. **Продажи и клиенты** — монетизация
7. **Расходы** — финансовый учёт
8. **Зарплата** — можно параллельно с продажами
9. **Отчёты** — после всех модулей, когда есть данные
10. **Уведомления** — финальная полировка

## Ключевые архитектурные решения, которые нужно принять рано

1. **Метод учёта себестоимости** — FIFO vs средневзвешенная цена
   - FIFO точнее, но сложнее в реализации
   - Средневзвешенная проще, подходит для малого бизнеса

2. **Партионный учёт или агрегированный**
   - Партионный: каждая закупка сырья — отдельная «партия» со своей ценой
   - Агрегированный: общий остаток со средней ценой

3. **Формат хранения денег** — decimal(18,2), никогда float

4. **Часовой пояс** — хранить в UTC, отображать в локальном

5. **Soft delete vs hard delete** — для финансовых данных только soft delete (пометка «удалено», но данные остаются)

## Риски проекта

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Неточная себестоимость из-за неучтённых расходов | Высокая | Высокое | Включить тару, потери, транспорт в расчёт |
| Расхождение остатков склада | Высокая | Среднее | Модуль инвентаризации + аудит лог |
| Сложность расчёта зарплаты (налоги) | Средняя | Среднее | Определить scope: с налогами или без |
| Рост объёма данных | Низкая | Среднее | Архивация + индексы БД |
| Изменение рецептур ломает историю | Средняя | Высокое | Версионирование рецептов |
